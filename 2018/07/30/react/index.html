组件的定义

组件的引入

如果你在代码中使用了 JSX 语法，那么你一定要在文件的顶部引入 react. 不引入 react 是没有办法编译 JSX 语法的。

最基本的 JSX 语法

在 javascript 中写 html 语法就是 JSX . 在 React 中必须使用 JSX 语法。

自定义的组件名必须以大写字母开头。

render 函数返回的html代码最外层必须是只有一个元素。这一点和 Vue 的 template
很像。不过不想最外层有一个包裹器，可以这样做：

React 中的响应式设计思想和事件绑定

react 中的组件的数据

react 不允许我们直接修改 state 中的内容。必须使用 this.setState 来修改。state 中的数据不允许修改，只能重置。

JSX 语法细节点

写注释：

用 className 代替 class ，即 css 的类名。

直接操作 innerHtml 用： dangerouslySetInnerHTML = { {__html: item} }

label 标签的 for 属性是用来扩到input框的点击范围的。即点击 label 时也可以 focus input框。不过在 react 中，for 这个关键字被认为是循环的那个 for 。建议 label 上面使用 htmlFor

组件间传值

父组件通过绑定属性的方式来向子组件传数据。

父组件通过向子组件传递一个方法，自组件调用这个方法就可以向父组件传递数据。

key 值应该放在组件的最外层上面。

单向数据流

父组件可以向子组件传值。子组件可以使用这个值，但不允许修改父组件传过来的值，要修改必须调用父组件的方法。父组件传过来的值是只读的。

PropTypes 和 DefaultProps

参数类型和默认参数。

PropTypes 限制父组件传给自组件的数据类型。

PropTypes.oneOfType( [ PropTypes.number, PropTypes.string ] ) 即类型是number和string 中的一个。

Props, State 与 render 函数

当组件的 props 或 state 发生变化时，render 函数就会重新执行。这样页面就跟着数据发生变化。

当父组件的 render 函数运行时，它的所有自组件的 render 函数都会被重新运行一次。

虚拟 DOM

虚拟 DOM 就是一个 JS 对象，用它来描述真实DOM。例如：

第一个为真实DOM结构，第二个为虚拟DOM结构。

基于虚拟DOM做 diff 算法更快。

虚拟DOM能这么块的主要原因是：

  • 每次比对之后再渲染，只渲染必须的那部分，提高的效率
  • 比对的是虚拟DOM，而不是真实DOM，JS 生成虚拟DOM很快，生成真实DOM由于要调用底层的api，比较慢。

虚拟DOM减少了真实DOM的创建以及真实DOM间的对比。极大的提升了性能。

JSX 模板 –> JS 对象( 即虚拟DOM ) –> 真实的DOM

JSX 模板 –> JS 对象( 即虚拟DOM ) ： React.createElement('div', {}, 'content')

通过 React.createElement 完全可以取代 JSX 语法，不过还是用 JSX 书写更加的简洁。

diff 算法

为了提升 React 底层的性能。setState 实质上是一个异步操作。如连续调用三次 setState 会自动把三次的调用合并为一次。

react 采用的是同层比对的算法。顶层不同，就不会看下面的几层。直接替换调。

key 值的使用可以提高性能，明确的知道，现在的虚拟DOM中的某个元素对应的以旧的虚拟DOM中的哪个元素。防止 react 的瞎复用。故 key 值的取名很重要。以前得名字，到新的虚拟DOM中不能变，这样才能起到作用。

key 值不要为 index 下标。key 值必须是一个稳定的值。

一个数组列表有两项，现在在第一项和第二项之间插入了第三项。[1, 2] –> [1, 3, 2] ，渲染时，既可以看做是第二项内容变为 3，新增第3项，内容为2。也可以看做是原来的第一项和第二项之间插入第三项，内容为3。key 的作用就是帮助 react 决定哪些元素应该复用。

ref 的使用

ref 即 reference 引用。用来操作DOM。

ref 后面写一个函数。

setState 是一个异步函数，事实上可以接受两个参数，第一个参数是你要修改的state 。第二个是可选参数，异步完成后的回调函数。这个回调函数里面用 ref 就能获取到更新后的真实DOM。

通常不建议过多使用 ref。

生命周期函数

生命周期函数指的是在某一个时刻组件会自动调用执行的函数。

componentWillMount 在组件即将被挂载到页面时刻执行。组件即将挂载

componentDidiMount 在组件刚刚被挂载到页面时刻执行。组件已经被挂载

render 页面渲染

shouldComponentUpdate 在组件更新前执行。返回 true 则会进行组件更新，返回 false 则组件更新被阻断。

componentWillUpdate 在组件render 之前执行，在组件 shouldComponentUpdate 之后执行。

componentDidUpdate 在组件 render 之后执行。

componentWillUnmount 在组件卸载前调用。

componentWillReceiveProps 的触发条件：

  • 一个组件要从父组件接受参数。
  • 只要父组件的 render 函数重新执行了，子组件的这个生命周期函数就会执行。
  • 如果这个组件第一次存在于父组件中，不会执行。
  • 如果这个组件已经存在于组件中，才会执行。、

子组件 render 被触发的条件：

  • 子组件的 state 或 props 发生变化了。

  • 父组件 render , 子组件的 render 也会被触发。（会带来性能损耗）

    可以通过 shouldComponentUpdate 来把由于父组件 render 触发的自组件的不必要的 render 给干掉。提高性能。

避免子组件做一些无谓的 render 操作。

AJAX 请求一般放在 componentDidMount() 阶段来做。当然你想在 componentWillMount() 阶段也可以，不过这个阶段发 AJAX 请求，不被 react Native 等高级的技术栈支持。所以，AJAX 请求最好放在 componentDidMount 阶段。

React 框架并没有发 AJAX 请求的能力，推荐使用 axios 来实现。

Redux 相关

Redux = Reducer + Flux

上图中，react components 相当于借阅者，action creators 相当于借阅者说的一句话：我要借 XXX 这本书。store 相当于图书馆管理员。 reducers 相当与图书馆管理员的图书借阅小册子，方便查阅图书还剩几本，哪些人借走了。

redux 的使用场景

  • 用户的使用方式复杂
  • 不同身份的用户有不同的使用方式（比如普通用户和管理员）
  • 多个用户之间可以协作
  • 与服务器大量交互，或者使用了WebSocket
  • View要从多个来源获取数据

从组件角度看，如果你的应用有以下场景，可以考虑使用 Redux。

  • 某个组件的状态，需要共享
  • 某个状态需要在任何地方都可以拿到
  • 一个组件需要改变全局状态
  • 一个组件需要改变另一个组件的状态

store

Store 就是保存数据的地方，你可以把它看成一个容器。整个应用只能有一个 Store。

Redux 提供createStore这个函数，用来生成 Store。

state

Store对象包含所有数据。如果想得到某个时点的数据，就要对 Store 生成快照。这种时点的数据集合，就叫做 State。

当前时刻的 State，可以通过store.getState()拿到。

Action

State 的变化，会导致 View 的变化。但是，用户接触不到 State，只能接触到 View。所以，State 的变化必须是 View 导致的。Action 就是 View 发出的通知，表示 State 应该要发生变化了。

Action 是一个对象。其中的type属性是必须的，表示 Action 的名称。其他属性可以自由设置.

可以这样理解，Action 描述当前发生的事情。改变 State 的唯一办法，就是使用 Action。它会运送数据到 Store。

React 中拆分 UI 组件和容器组件

UI 组件专门做渲染，容器组件专门做逻辑。

无状态组件

当一个组件里面只有一个 render 函数时，就可以称之为 无状态组件

无状态组件的性能更高，如果一个组件里面只有 render 函数，那么就应该把它改写成无状态组件。

无状态组件性能高是因为它只需要返回一个JSX 块，不需要走那些声明周期函数

UI 组件通常要写成无状态组件。

中间件 middleware

Redux 的基本做法：用户发出 Action ，Reducer 函数算出新的 State, View 重新渲染。

Action 发出以后，Reducer 立即算出 State，这叫做同步；Action 发出以后，过一段时间再执行 Reducer，这就是异步。

怎么才能 Reducer 在异步操作结束后自动执行呢？ 这就要用到新的工具：中间件（middleware）

中间件就是一个函数，对store.dispatch 方法进行了改造，在发出 Action 和执行 Reducer 这两步之间，添加了其他功能。

把你需要的中间件放在 applyMiddleware 方法之中，传入 createStore 方法，就完成了 store.dispatch() 的功能增强。

异步操作的基本思路

理解了中间件以后，就可以处理异步操作了。

  • 操作发起时的 Action
  • 操作成功时的 Action
  • 操作失败时的 Action

Immutable.js

Immutable.js 运用到 redux 中，可以防止 state 被随意修改。

生成一个immutable 对象： var immutableObject = fromJS({a:1})

取 immutable 对象中的值： immutableObject.get('a')

设置 immutable 对象的值： immutableObject.set('a',2)

immutable 对象的 set 方法，会结合该对象之前的值和要设置的值，返回一个全新的immutable 对象。immutable 对象永远不会被修改，只会返回新的。

什么是路由

路由，即根据 url 的不同，显示不同的页面。

纯组件

PureComponent 就是纯组件。通常 PureComponent 和 Immutable.js 配合起来使用。不然会有一些坑。

有了 PureComponent ，我们就不用每次写 shouldComponentUpdate 了。PureComponent 可以自动处理，只有我这个组件引用的相关 store 中的数据发生变化了，才会重新执行 render 函数，而不是以前那样，只要 store 中的数据发生了变化，全局的所有组件都重新渲染。

react-loadable

实现单页应用，只有在进入某个页面的时候，才加载这个页面对应的代码。

这样可以首页加载时把这个应用的文件全部加载进来。react-loadable 的使用可以大大的减少首屏的加载时间。

这样就把以前的普通组件，变成了异步组件。实际要加载的组件被包装成了 LoadableComponent 组件。路由跳转时对应的组件应该变成 包装后的组件LoadableComponent 。不过这也可能导致原来的路由id, 即 this.props.match.params.id ，LoadableComponent 组件可以拿到，而真正被包装的组件拿不到。可以通过 withRouter 来解决这个问题。
